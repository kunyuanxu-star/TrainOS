## Overview

Wave 1 implements three independent versions in parallel, each forming the base of one development track:

| Version | Track | Theme |
|---------|-------|-------|
| V21 | A (Security) | Formal verification & security hardening |
| V22 | B (Performance) | High-performance async I/O (io_uring) |
| V23 | C (Ecosystem) | Virtualization & hypervisor |

These three have no mutual dependencies and can be developed concurrently. Each version's syscalls use a reserved number range to avoid merge conflicts in the dispatch table.

## V21 — Formal Verification & Security Hardening

### Current State

`kernel/src/security/mod.rs` (~200 lines): W^X enforcement, seccomp filter shell, cap audit buffer, stack canary.
`kernel/src/invariant.rs` (~99 lines): Memory/scheduler/IPC/W^X invariant placeholders.

### 21.1 Kernel Invariant Formalization

**File**: `kernel/src/invariant.rs`

#### 21.1a Scheduler Invariant
```
For each set bit i in priority_bitmap:
  ready_queues[i] must contain >= 1 thread
  Each thread in ready_queues[i] must have state == Ready
  Each thread's effective priority must equal i
```
- Walk `priority_bitmap`; for each set bit, iterate the corresponding `ready_queues[i]`
- If empty queue found with set bit → panic with "INVARIANT: scheduler bitmap/queue mismatch"
- If thread state != Ready → panic with pid + actual state

#### 21.1b Memory Invariant
```
For each free page in buddy free_list[order]:
  ref_count must be 0
sum(allocated_pages) + sum(free_pages) == total_physical_pages
```
- Count allocated pages via page table walk (all user PTEs with V=1)
- Count free pages via buddy free lists (all orders)
- Compare sum against `TOTAL_PHYSICAL_PAGES` constant

#### 21.1c IPC Invariant
```
For each endpoint's wait_queue:
  Detect cycles using Floyd's algorithm
  No thread may appear in the queue twice
```
- Use two-pointer algorithm on the linked list
- Check for duplicate thread pointers

#### 21.1d Periodic Check Trigger
**File**: `kernel/src/trap/mod.rs`
- Add counter `invariant_check_counter: u64` in per-CPU data
- Increment on each timer interrupt; when `counter % 100 == 0`, call `run_all_invariants()`
- `run_all_invariants()` calls each check function sequentially; panics kernel on failure

### 21.2 Capability Security

#### 21.2a sys_mint Deep Validation
**File**: `kernel/src/syscall/cap.rs`
- Before mint: load parent cap rights, compute child rights
- Runtime assertion: `(child_rights & !parent_rights) == 0`
- On failure: return error code, log to audit buffer

#### 21.2b Cap Audit Log Enhancements
**File**: `kernel/src/security/mod.rs`
- Audit entry: `{ pid: u32, op: u8, slot: u32, timestamp: u64 }`
- 64-entry circular buffer → expand to 256 entries
- `sys_cap_audit_read(buf: *mut u8, count: *mut u32)` — copy entries to user buffer
- `/proc/cap_audit` virtual file for human-readable dump

#### 21.2c Cap Leak Detection on Process Exit
**File**: `kernel/src/proc/mod.rs`
- In `kill_process()`: iterate the process's CNode, count unreleased caps
- If count > 0: write to audit log "PID X leaked N capabilities"
- Auto-revoke all remaining caps via `revoke_all(process.cnode_id)`

#### 21.2d /proc/cap Virtual File
**File**: `services/proc/` (procfs service)
- Read returns: one line per capability — `slot type rights cnode`
- Example: `3 Endpoint 0x1F 7`

### 21.3 Memory Safety

#### 21.3a Heap Canary
**File**: `kernel/src/mem/heap.rs`
- Each allocation: prepend 8-byte canary `0xDEAD_BEEF_CAFE_BABE`, append same canary after payload
- `kernel_free()`: check both canaries before deallocating; panic on mismatch
- Canary value stored as `const` — no runtime generation needed (no ASLR in kernel heap)

#### 21.3b User Buffer Bounds Checking
**File**: `kernel/src/syscall/posix.rs`
- In `sys_read`/`sys_write`: before copying data, validate `buf..buf+len` is within user address space
- Use `is_user_range_valid(vaddr, len) → bool` helper in `mem/sv39.rs`
- Walk page table for the range; every page must have U=1 and accessible

#### 21.3c W^X Auto-Fix
**File**: `kernel/src/mem/sv39.rs` + `kernel/src/trap/mod.rs`
- `verify_wxorx()` walks all user PTEs; any page with both W=1 and X=1 → violation
- `force_wxorx()`: for each violation, clear the X bit (prefer data over code for safety)
- Call `force_wxorx()` on process spawn and on page fault if W^X violation detected
- Log each fix: "W^X: cleared X on vaddr 0x... for PID N"

#### 21.3d Stack Guard Page
**File**: `kernel/src/mem/layout.rs` + `kernel/src/trap/mod.rs`
- Each kernel stack: 64KB total, last 4KB = guard page (unmapped)
- In trap handler: if fault address is within guard page of current kernel stack → stack overflow
- Response: kill current process (not kernel panic, as this could be user-triggered via deep recursion)

### 21.4 Syscall Audit

#### 21.4a Seccomp Filter Enhancement
**File**: `kernel/src/security/mod.rs`
- Current: single rule per process → enhance: up to 32 rules per process
- Rule: `{ syscall_nr, action: Allow | Kill | Log }`
- Match logic: first matching rule wins; if no rule matches, default-allow
- `sys_seccomp_add(syscall_nr, action)` adds rule for calling process

#### 21.4b /proc/syscalls
**File**: `kernel/src/syscall/mod.rs` + `services/proc/`
- Kernel: `SYSCALL_COUNTERS: [u64; 256]` global array
- Increment on each syscall dispatch (before seccomp check)
- `/proc/syscalls` reads: one line per non-zero counter — `nr name count`
- Reset via `sys_syscall_stats_reset()` (syscall 132)

#### 21.4c Sensitive Op Audit
**File**: `kernel/src/security/mod.rs`
- Hooking points in `sys_kill`, `sys_mmap`, `sys_munmap`, `sys_mprotect`
- Each logs: `{ pid, syscall_nr, arg_summary }` to audit buffer
- arg_summary for mmap: addr+len; for kill: target_pid+sig; for mprotect: addr+prot

### V21 Acceptance Criteria
- All 4 invariant checks pass on boot and periodic runs
- Cap audit log readable via `/proc/cap_audit`
- W^X enforced: no page is both writable and executable
- Heap double-free or overflow detected by canary
- Seccomp filter blocks denied syscalls

## V22 — High-Performance Async I/O (io_uring)

### Current State

`kernel/src/iouring/mod.rs` (~155 lines): Ring struct, SQE/CQE definitions, submit/peek stubs.

### 22.1 io_uring Core

#### 22.1a SQE/CQE Structure Completion
**File**: `kernel/src/iouring/mod.rs`
```rust
pub struct SQE {
    pub opcode: u8,       // IORING_OP_*
    pub flags: u8,        // IOSQE_FIXED_FILE etc.
    pub fd: u32,          // target file descriptor
    pub addr: u64,        // user buffer address
    pub len: u32,         // buffer length
    pub off: u64,         // file offset
    pub user_data: u64,   // returned in CQE
}

pub struct CQE {
    pub user_data: u64,   // from SQE
    pub res: i32,         // result code or bytes transferred
    pub flags: u32,       // completion flags
}
```

#### 22.1b Ring Shared Memory Mapping
**File**: `kernel/src/iouring/mod.rs` + `kernel/src/mem/sv39.rs`
- `sys_io_uring_setup(entries: u32)`:
  1. Allocate 2 contiguous physical pages: one for SQ, one for CQ
  2. Each ring: array of `entries` SQEs/CQEs + head/tail indices
  3. Map both pages into caller's address space (user-accessible)
  4. Store ring metadata in kernel's `RINGS` table
  5. Return ring fd

#### 22.1c Real I/O Submission
**File**: `kernel/src/iouring/mod.rs`
- `submit(ring_id)`: iterate SQ from `sq_head` to `sq_tail`
- For each SQE, dispatch based on opcode:
  - `IORING_OP_NOP` → post CQE with res=0
  - `IORING_OP_READ` → forward to VFS read (EP 2), post CQE with byte count
  - `IORING_OP_WRITE` → forward to VFS write, post CQE with byte count
  - `IORING_OP_OPEN` → call VFS open/create, post CQE with fd
  - `IORING_OP_CLOSE` → per-process fd table close, post CQE
  - `IORING_OP_STAT` → VFS stat, post CQE
- Update `sq_head` after processing
- VFS IPC is synchronous within kernel, so CQE is posted immediately

#### 22.1d Completion Peek
**File**: `kernel/src/iouring/mod.rs`
- `peek_cqe(ring_id, cqe_ptr: *mut CQE)`: copy head CQE to user buffer, advance `cq_head`
- Return number of CQEs available

### 22.2 Zero-Copy Data Path

#### 22.2a Shared Memory Page Passing
**File**: `kernel/src/mem/sv39.rs`
- `share_page(src_pid, dst_pid, src_vaddr, dst_vaddr)`:
  1. Look up physical page of `src_vaddr` in src process page table
  2. Map same physical page into dst process at `dst_vaddr` with same permissions
  3. Increment page refcount
- Used by io_uring for I/O buffers larger than 64 bytes

#### 22.2b VM_SHARED Enhancement
**File**: `kernel/src/syscall/memory.rs`
- `sys_shm_map` already creates shared mappings; add `sys_shm_splice(src, dst, offset, len)` to transfer pages between shared regions without copy
- Implementation: remap physical pages from one shared region to another

### 22.3 Block Device Optimization

#### 22.3a Request Merging
**File**: `kernel/src/device/mod.rs` (new: `device/merge.rs`)
- Before submitting to VirtIO queue: sort pending requests by sector number
- Merge adjacent requests: if req_a.end_sector >= req_b.start_sector, combine into single DMA operation
- Max merge segments: 128 (VirtIO limit)

#### 22.3b Multi-Queue (blk-mq)
**File**: `kernel/src/device/mod.rs`
- Per-CPU submission queue: `BLK_QUEUES: [BlkQueue; MAX_CPUS]`
- Each CPU submits to its own queue → no lock contention
- Single completion IRQ handler dispatches to originating CPU

#### 22.3c I/O Scheduler Framework
**File**: `kernel/src/device/sched.rs` (new)
```rust
pub trait IoScheduler {
    fn enqueue(&mut self, req: BlockRequest);
    fn dequeue(&mut self) -> Option<BlockRequest>;
    fn name(&self) -> &str;
}
```
- `NoopScheduler`: FIFO, no reordering
- `DeadlineScheduler`: read requests get 500ms deadline, write requests 5s; expired requests jump queue

### V22 Acceptance Criteria
- io_uring read/write throughput > 2x synchronous read/write (measured by bench service)
- Large file transfer latency < 50% of synchronous IPC (4KB+ buffers)
- Request merging reduces VirtIO operations by >20% on sequential I/O
- blk-mq: per-CPU queue utilization shows balanced distribution

## V23 — Virtualization & Hypervisor

### Current State

`kernel/src/hypervisor/mod.rs` (~63 lines): VM create/destroy/start stubs, VM list.

### 23.1 Hypervisor Mode

#### 23.1a H-Extension CSR Operations
**File**: `kernel/src/hypervisor/csr.rs` (new)
- Wrap RISC-V H-extension CSRs in safe Rust functions:
  - `hgatp_read()/write()` — guest address translation table
  - `hstatus_read()/write()` — hypervisor status (VSBE, GVA, etc.)
  - `hedeleg_read()/write()` — exception delegation
  - `hideleg_read()/write()` — interrupt delegation
- Use inline assembly with `.insn` directives

#### 23.1b Two-Stage Address Translation
**File**: `kernel/src/hypervisor/mmu.rs` (new)
- G-stage (Guest physical → Host physical) page table management
- `create_gstage_page_table(vm_id)`: allocate root page for VS-mode
- Map guest physical pages to host physical pages
- Return `hgatp` value for CSR write

#### 23.1c VM Lifecycle
**File**: `kernel/src/hypervisor/mod.rs`
- `vm_create(name, memory_mb)`: allocate physical memory from buddy allocator, create G-stage PT, assign vmid
- `vm_destroy(vmid)`: free all guest memory, destroy G-stage PT, remove from VM table
- `vm_start(vmid, entry_pc)`: set up vCPU CSRs (sepc ← entry_pc), configure `hgatp`, enter VS-mode via `sret`
- `vm_pause(vmid)`: inject external interrupt to vCPU, return to host
- `vm_resume(vmid)`: restore CSRs, `sret` back to guest

### 23.2 Virtual I/O

#### 23.2a VirtIO Backend
**File**: `kernel/src/hypervisor/virtio.rs` (new)
- Guest VirtIO MMIO reads/writes trapped by hypervisor
- Decode VirtIO device ID, queue, descriptor
- Forward to host-side service via IPC (EP for block → drv service, EP for net → veth service)
- Return guest VirtIO completion via interrupt injection

#### 23.2b Paravirtual Timer
**File**: `kernel/src/hypervisor/timer.rs` (new)
- Guest reads `time` CSR → trapped, hypervisor returns host time + offset
- Guest sets timer compare → trapped, hypervisor programs host timer
- On host timer fire → inject timer interrupt into guest

#### 23.2c Virtual Interrupt Controller
**File**: `kernel/src/hypervisor/plic.rs` (new)
- Emulate PLIC (Platform-Level Interrupt Controller) for each VM
- Host interrupts routed to owning VM via virtual PLIC
- Guest reads `claim`/writes `complete` → trapped and forwarded to host PLIC

### 23.3 Lightweight VMs

#### 23.3a Single-Kernel Unikernel Mode
**File**: `kernel/src/hypervisor/unikernel.rs` (new)
- Guest runs same TrainOS kernel binary, no separate OS image needed
- Guest kernel ELF loaded directly into VM memory
- Minimal device tree for guest (just serial + timer)

#### 23.3b Snapshot/Restore
**File**: `kernel/src/hypervisor/snapshot.rs` (new)
- `vm_snapshot(vmid) → Vec<u8>`:
  1. Pause vCPU
  2. Serialize: all CSRs (sepc, sstatus, stvec, etc.) + GPRs (x1-x31) + guest physical memory (all allocated pages)
  3. Write to buffer, return to caller
- `vm_restore(vmid, snapshot_data)`: reverse process, resume vCPU
- Snapshot size ≈ VM memory size + 4KB metadata

### V23 Acceptance Criteria
- Can run a nested TrainOS instance inside a VM
- Snapshot and restore completes correctly (guest resumes without visible interruption)
- VirtIO block/net forwarding works (guest can read/write disk via host driver service)

## Cross-Cutting Concerns

### Shared Code Coordination

To minimize merge conflicts across parallel V21/V22/V23 development:

1. **`kernel/src/syscall/mod.rs`**: Each version appends its syscall entries within its reserved number block. Mergeable by concatenation.
2. **`lib/tros/src/lib.rs`**: Same — syscall constants added in reserved ranges.
3. **`kernel/src/main.rs`**: Each version adds its `mod` declaration and init call in a labeled section.
4. **New files** (e.g., `iouring/`, `hypervisor/` subdirectories): No overlap risk.

### Build Verification

After each version is implemented, and after Wave 1 merge:
```bash
make all    # must compile with 0 errors
make test   # all 22+ self-tests must pass
```

### Documentation Updates

Each version updates:
- `CLAUDE.md`: version status, new files in table, new syscall count
- `README.md` (if present): overall version milestone
- Roadmap file: mark completed checkboxes with `[x]`

## Implementation Order

Within Wave 1 (V21, V22, V23 can be implemented in any order):

1. **V21 first** (recommended) — security foundation benefits later work
2. **V22 and V23** — parallel after V21 complete

Or truly parallel: dispatch all three to independent agents simultaneously.
